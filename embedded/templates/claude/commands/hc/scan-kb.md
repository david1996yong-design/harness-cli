扫描代码库，在 `.harness-cli/kb/prd/` 下生成产品知识库文档。

执行以下步骤：

1. **确认 KB 目录存在**：
   检查 `.harness-cli/kb/prd/` 是否存在。若不存在，提示用户先运行 `harness-cli scan`，然后停止。

2. **读取模块模板**：
   ```bash
   cat .harness-cli/kb/prd/_module-template.md
   ```
   这是每个模块文档需要遵循的格式。

3. **读取现有索引**，了解当前状态：
   ```bash
   cat .harness-cli/kb/prd/index.md
   ```

4. **分析项目结构**：
   - 梳理顶层目录结构
   - 识别入口点（main 文件、路由、控制器、处理器）
   - 查找模型/Schema/类型定义
   - 查找配置和依赖文件

5. **识别业务模块**：
   根据以下维度将相关源文件归类为逻辑业务模块：
   - 目录组织（每个顶层 src 目录通常对应一个模块）
   - 命名模式（如 `user_*`、`auth_*`、`payment_*`）
   - 领域边界（协作完成同一功能的文件）
   - 公共 API 接口（导出函数、CLI 命令、路由处理器）

6. **生成模块文档**：
   对每个识别出的模块：
   - 阅读关键源文件以理解业务逻辑
   - 按模板格式创建 `.harness-cli/kb/prd/<模块名>.md`
   - 记录：用途、功能、业务规则、数据流、边界情况、依赖关系、关键文件
   - 记录代码**实际做了什么**，而非理想状态

7. **更新索引**：
   更新 `.harness-cli/kb/prd/index.md` —— 在模块索引表中填入所有已发现的模块。

8. **输出摘要**：
   列出所有已记录的模块，每个模块附一句话描述。

**重要事项**：
- 所有文档使用**中文**撰写
- 专注于业务逻辑和产品行为，**不要**写编码规范（那属于 `spec/`）
- 如果是重新扫描，先读取现有模块文档，更新而非覆盖
- 对于大型项目，优先扫描最重要的模块，其余标注留待后续
