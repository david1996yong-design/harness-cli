# Tech 文档模板与写作指引

> 本文件定义 `kb/tech/` 下各固定文档的结构与写作要求。

---

## overview.md -- 系统全景

记录系统的整体技术面貌，帮助 AI agent 快速理解「这个系统用了什么、长什么样」。

### 结构要求

```markdown
# 系统全景

## 技术栈

| 类别 | 技术 | 版本 | 说明 |
|------|------|------|------|
| 语言 | ... | ... | ... |
| 框架 | ... | ... | ... |
| 构建工具 | ... | ... | ... |
| 包管理 | ... | ... | ... |
| 测试 | ... | ... | ... |

## 核心组件

| 组件 | 一句话描述 | 入口文件 |
|------|-----------|----------|
| ... | ... | ... |

## 系统边界

- 外部依赖（API、第三方服务）
- 输入输出接口
- 部署目标
```

## component-map.md -- 组件关系

记录组件之间的调用关系、数据流向、依赖方向。

### 结构要求

```markdown
# 组件关系

## 依赖关系图

用文字或 ASCII 描述组件间的依赖方向。

## 调用链

| 调用方 | 被调用方 | 方式 | 说明 |
|--------|---------|------|------|
| ... | ... | 函数调用 / 事件 / HTTP | ... |

## 数据流

描述核心数据从输入到输出的流转路径。

## 依赖方向原则

说明项目遵循的依赖方向规则（如：单向依赖、分层架构等）。
```

## decisions.md -- 架构决策记录（ADR-lite）

记录重要的技术选型和架构决策，包括背景、选项和最终决定。

### 结构要求

```markdown
# 架构决策记录

## ADR-001: {决策标题}

- **状态**: 已采纳 / 已废弃 / 讨论中
- **日期**: YYYY-MM-DD
- **背景**: 当时面临什么问题
- **选项**:
  1. 选项 A -- 优点 / 缺点
  2. 选项 B -- 优点 / 缺点
- **决定**: 选择了什么，为什么
- **后果**: 这个决定带来了什么影响
```

```markdown
- 配置优先级
- 配置验证

## 共享工具函数

| 工具 | 文件位置 | 用途 |
|------|----------|------|
| ... | ... | ... |

## 中间件 / 拦截器

（如有）描述请求处理管道中的中间件链。
```
